1. Tổng quan trách nhiệm

Sensor state → Web cập nhật knownMaze → FF / FF2 / DK / DK2
Web gửi action hoặc primitive queue → Mouse thực thi → trả trạng thái

Web chịu trách nhiệm

  • Lưu knownMaze, vị trí và heading.
  • Tính route, cost và proof confirmed/optimistic.
  • Trong explore chỉ gửi action 4 hướng.
  • Sau proof gửi route/primitive queue one-way.

Mouse chịu trách nhiệm

  • Điều khiển motor, encoder, gyro và wall follower.
  • Thực thi action/primitive đúng thứ tự.
  • Đọc tường theo heading hiện tại.
  • Emergency stop khi stall hoặc sai pose.

Boundary: mouse planners không đọc trueMaze, chỉ đọc knownMaze. Trong simulator, SimulatedMouse.readCardinalSensors(), truth canvas và truth analysis được phép dùng maze thật.

2. Giải thuật từ khám phá đến speed run

Ranh giới dữ liệu: trueMaze chỉ dành cho simulated sensor và truth analysis/canvas. Mouse planner không nhìn maze thật; mọi route và proof phục vụ mouse đều được tính từ knownMaze do sensor cập nhật.

Goal/target = 0 Mouse Ô kế tiếp n - 1 Ứng viên frontier

buildDistanceMap() là BFS nhiều nguồn. Ban đầu mỗi target = 0 được đưa vào queue. Khi lấy một ô mang số n, mọi láng giềng đi được và chưa có số nhận neighbor = n + 1.

Nếu target là vùng đích, bốn ô goal cùng mang 0 và cùng được đưa vào queue. Vì BFS lan theo từng lớp, con số trong ô là số bước cardinal ngắn nhất tới target trong graph đang xét — không phải số lần mouse đã đi.

Tường đã xác nhận luôn chặn cạnh. Khi tính optimistic hoặc đường tới frontier, cạnh unknown có thể được giả sử là mở; vì thế sensor mới có thể làm các số và route thay đổi.

queue = tất cả target
distance[target] = 0
while queue chưa rỗng:
  current = queue.popFront()
  for neighbor đi được và chưa có số:
    distance[neighbor] = distance[current] + 1
    queue.pushBack(neighbor)

Mouse policy: đi theo số giảm dần

followDistanceMap() đang ở ô n chỉ nhận ứng viên có neighbor = n - 1. Nếu nhiều ô cùng đạt số thấp nhất, heading hiện tại tạo tie-break cố định: forward → turnLeft → turnRight → turnAround. Vì vậy route không random khi hai phía có cùng số.

Trong API cardinal, quay không đổi position; forward mới đổi position sang ô kế tiếp. Do đó để rẽ rồi sang ô bên cạnh, mouse thực thi một action quay, sau đó một action forward. Sensor cập nhật knownMaze thì distance map được tính lại trước quyết định tiếp theo.

Nếu số tại mouse là ∞, hoặc không có láng giềng đúng n - 1, route được xem là không hợp lệ; policy không tự xuyên tường hay chọn hướng ngẫu nhiên.

candidates = neighbor đi được có distance = n - 1
sort candidates theo forward, turnLeft, turnRight, turnAround
next = candidates[0]
if heading chưa trùng hướng next: quay tại chỗ
else: forward sang next

Frontier: chọn nơi cần sensor xác nhận

Một ô là ứng viên frontier khi còn ít nhất một wall state = null. Nó không đơn giản là “ô chưa từng ghé”, không phải goal cuối và cũng không đồng nghĩa với ô đang hiện ∞.

Policy ưu tiên frontier trên optimistic route của planner đang học. Phần fallback được sắp ổn định theo distance-to-route → distance-to-mouse → row → col. Hệ thống duyệt theo thứ tự đó và lấy ứng viên đầu tiên có route reachable từ mouse.

Frontier được chọn trở thành target = 0; BFS đánh lại toàn distance map với cạnh unknown được phép giả định mở. Mouse vẫn tới đó bằng route/action 4 hướng. Tại frontier, sensor ghi tường vào knownMaze, sau đó proof và danh sách frontier được tính lại.

optimistic route → frontier có wall null → kiểm tra reachable → frontier = 0 → BFS → mouse theo số giảm → sensor → knownMaze/proof mới 
preferred = ô wall null trên optimistic route
fallback = ô wall null sắp theo distance-to-route,
           distance-to-mouse, row, col
frontier = ứng viên đầu tiên reachable
distance = FloodFill(target = frontier, allowUnknown = true)
mouse đi 4 hướng theo distance giảm dần rồi đọc sensor

Flood Fill (FF)

State là (cell, heading) trên graph 4 hướng. FF chọn đường theo objective lexicographic (steps, turns): ít bước trước, rồi mới ít lượt quay; tie còn lại dùng thứ tự canonical cố định. Khi explore, mouse chỉ thực thi cardinal action ngang/dọc.

FF2

Vẫn là graph 4 hướng, nhưng state giữ chiều dài đoạn thẳng hiện tại. Với r = √0.5, chi phí một đoạn dài x là costM(x) = 0.5x + 0.5(1-r^x)/(1-r); mỗi 90° quay cộng π/4. Objective là (cost, steps, turns), rồi tie-break canonical.

Dijkstra 8 hướng (DK)

Node là pose tại gate-point gồm vị trí, heading8 và trạng thái diagonal lane. Chỉ cạnh motion primitive vượt qua full clearance với tường/post mới được thêm. Cost từ source: MH=0.5, M=1, MD=√0.5, R1/L1=0.5+√0.5, R/L=1+√0.5.

DK2

DK2 dùng cùng graph, pose, primitive và clearance của DK, nhưng thêm mRunLength vào state. Cạnh M thứ x có deltaM(x) = costM(x) - costM(x-1); primitive khác reset đoạn về 0 và giữ cost DK. Đây là một lần chạy Dijkstra độc lập, không phải lấy route DK rồi rescore.

Proof: confirmed và optimistic

Mỗi planner chạy hai lần trên knownMaze: confirmed coi cạnh unknown là bị chặn, còn optimistic cho phép đi qua unknown để tạo cận dưới. Chỉ khi cả hai hợp lệ, objective bằng nhau trong epsilon và canonical signature của route cũng giống nhau thì planner mới được chứng minh; bằng cost nhưng khác route chưa đủ.

Lifecycle

sensor → cập nhật knownMaze → chọn frontier trên route optimistic → đi explore bằng cardinal action → tính lại proof → về start → chạy one-way/speed run bằng route confirmed

FF, FF2, DK và DK2 được chứng minh tuần tự. Trong lúc học, mouse chỉ đi 4 hướng tới frontier cần quét; primitive chéo của DK/DK2 chỉ được gửi ở lượt speed run sau proof.

3. executeCardinalAction(request)

Dùng trong explore và điều khiển FF thủ công. Mỗi request chỉ là một action 4 hướng. Quay diễn ra tại ô hiện tại; forward mới thay đổi position.

Request từ web

{
  "action": "forward | turnLeft | turnRight | turnAround",
  "nextPosition": { "row": 5, "col": 8 },
  "durationMs": 120,
  "revision": 42
}

Response từ mouse

{
  "ok": true,
  "cancelled": false,
  "action": "forward",
  "moved": true,
  "position": { "row": 5, "col": 8 },
  "heading": "right",
  "completedAt": 1784200000000
}

revision hủy response animation/action cũ sau Reset. Hardware có thể ánh xạ nó thành command sequence ID.

4. readCardinalSensors(request)

Đọc sensor khi robot đứng trong tâm ô và đang nhìn theo một trong bốn heading. Hardware tối thiểu trả front/left/right; position và back là tùy chọn.

Request

{
  "heading": "top | right | bottom | left",
  "position": { "row": 5, "col": 8 }
}

Response tối thiểu

{
  "heading": "right",
  "walls": {
    "front": false,
    "left": true,
    "right": false
  },
  "position": { "row": 5, "col": 8 }
}

true nghĩa là có tường. Nếu trả thêm walls.back, web sẽ cập nhật đủ bốn cạnh. Nếu không có, cạnh sau giữ nguyên trạng thái đã biết/unknown.

5. executeMotionPrimitive(request)

Dùng cho DK/DK2 speed run sau proof. Planner đã kiểm tra clearance; mouse chỉ thực thi primitive queue liên tục với look-ahead.

MH
Nửa ô đầu từ tâm start tới gate.
M
Đi thẳng cardinal một đơn vị planner.
MD
Đi thẳng chéo giữa hai gate point.
R / L
Cua liên tục 90° gate-to-gate.
R1 / L1
Chuyển heading 45° vào hoặc ra diagonal.

Request primitive

{
  "type": "M | MD | R | L | R1 | L1 | MH",
  "from": { "x2": 1, "y2": 30, "headingIndex": 0 },
  "to":   { "x2": 2, "y2": 29, "headingIndex": 2 },
  "cost": 1.7071067812
}

Response

{
  "ok": true,
  "type": "R",
  "encoderMm": 127.3,
  "headingDeg": 90,
  "completedAt": 1784200000000
}

6. Gửi toàn route speed run

Web có thể gọi từng primitive như simulator, hoặc transport adapter gửi toàn packet để firmware look-ahead và chạy không dừng giữa action.

{
  "version": 1,
  "run_id": "dk2-20260717-001",
  "planner": "DK2",
  "cell_mm": 180,
  "start": { "row": 15, "col": 0, "heading": "top" },
  "primitives": [
    { "type": "MH", "count": 1 },
    { "type": "M", "count": 13 },
    { "type": "R1", "count": 1 },
    { "type": "MD", "count": 3 },
    { "type": "L1", "count": 1 }
  ],
  "checksum": "sha256:..."
}
  1. Mouse kiểm tra version, checksum, run_id và start pose.
  2. Giải nén queue, look-ahead để đặt entry/exit speed.
  3. Chạy liên tục; không đặt vận tốc về 0 giữa primitive.
  4. Dừng tại goal hoặc khi safety monitor báo lỗi.

7. Lỗi và an toàn

  • Emergency stop: motor stall, encoder/gyro lệch ngưỡng, mất wall reference hoặc bumper kích hoạt.
  • Response lỗi dùng { "ok": false, "error": "MOTOR_STALL" }.
  • Sai start pose/checksum/version: từ chối chạy, không tự sửa route.
  • run_id chống chạy lại packet cũ hoặc trùng lặp.
  • Khi executor lỗi, web hủy proof planner tương ứng và khóa các nút one-way.

8. Ví dụ HardwareMouse adapter

class HardwareMouse extends MouseAdapter {
  async executeCardinalAction(request) {
    return uartOrHttpSend("mouse/cardinal-action", request);
  }

  async readCardinalSensors(request) {
    return uartOrHttpSend("mouse/cardinal-sensors", request);
  }

  async executeMotionPrimitive(request) {
    return uartOrHttpSend("mouse/motion-primitive", request);
  }
}

// Thay đúng một dòng; planner và UI không đổi.
const mouseApi = new HardwareMouse();

Transport có thể là UART framing, HTTP JSON hoặc WebSocket. Adapter chịu trách nhiệm serialize/timeout/retry; planner chỉ thấy object contract ở trên.
